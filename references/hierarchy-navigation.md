# 階層ナビゲーションガイド

## 1. 階層構造の概要

```
Company（企業）
  └── Business（事業）
        └── Project（プロジェクト）
```

- **Company**: 法人または個人事業主の単位
- **Business**: 企業が営む事業ドメインの単位（EC事業、飲食事業、SaaS事業 等）
- **Project**: 事業計画の作成単位（補助金申請、新規事業立ち上げ、年度計画 等）

1つの企業が複数の事業を持ち、1つの事業が複数のプロジェクトを持つことができる。

---

## 2. ユーザーへの作業レベル確認方法

セッション開始時、ユーザーがどのレベルで作業したいかを確認する。

### 確認フロー

```
「今日はどのレベルで作業しますか？」

(a) 企業全体に関わること（企業情報の更新、新規事業の追加 等）
(b) 特定の事業に関わること（事業プロファイルの更新、財務データ取込 等）
(c) 特定のプロジェクトに関わること（事業計画の作成・更新 等）
```

ただし、ユーザーが具体的な作業内容を既に伝えている場合は、文脈から適切なレベルを推定し、確認を簡略化する。

---

## 3. セッション開始時の手順

### Step 1: `_active-context.md` を確認

前回のセッションで設定された対象（企業/事業/PJ）を確認する。

### Step 2: 継続か変更かを確認

```
「前回は [企業名] > [事業名] > [PJ名] で作業していました。今日も同じ対象で続けますか？」
```

### Step 3: 対象が変わる場合は `_active-context.md` を更新

変更後の対象を記録してからセッションを開始する。

---

## 4. 同一事業内でのプロジェクト切替

同じ事業の下にある別のプロジェクトに切り替える場合:

1. 現在のプロジェクトのセッション終了チェックリストを実行
2. `_active-context.md` の `project` を新しいスラッグに更新
3. 新しいプロジェクトの `_index.md` と `phase-state.md` を読み込む
4. 事業レベルのコンテキスト（`business-profile.md`, `financial-lens.md`）はそのまま保持

```markdown
# 切替前
- company: acme-corp
- business: ec-business
- project: subsidy-2025

# 切替後
- company: acme-corp
- business: ec-business
- project: expansion-2026
```

---

## 5. 事業コンテキストを継承する新規プロジェクト

既存事業の下に新しいプロジェクトを作成する場合:

1. `_template/project/` から新規プロジェクトディレクトリを作成
2. 事業の `_index.md` に新プロジェクトを追記
3. `_registry.md` に追記
4. `_active-context.md` を新プロジェクトに更新

新プロジェクトは以下の事業レベル情報を自動的に継承する:

- `business-profile.md`（事業概要、強み、市場環境）
- `financial-lens.md`（オーナーの財務の見方）
- `closing-status.md`（締めステータス）
- 事業レベルの財務データ

プロジェクト固有の情報（フェーズ、セッション履歴、意思決定）は新規に構築する。

---

## 6. クロスプロジェクトインサイト

同一事業内の異なるプロジェクト間で、知見を横展開する。

### 活用場面

- Project A で得た市場調査結果が Project B にも適用できる場合
- Project A の補助金申請で作成した財務モデルの前提が Project B でも使える場合
- Project A で判明したリスク要因が Project B にも影響する場合

### 運用ルール

- クロスプロジェクトの知見は事業レベルの `business-profile.md` に統合する
- プロジェクト固有の文脈に依存する知見は、参照元プロジェクトを明記した上で引用する
- 「Project A のセッション 2026-03-15 で判明した [内容]」のように出典を残す

---

## 7. クロスビジネスインサイト

同一企業内の異なる事業間で観察されるパターンを活用する。

### 活用場面

- EC事業と飲食事業で共通する顧客層の特徴
- 事業間で共有できるリソース（人材、設備、ブランド）
- 企業全体の財務状況が各事業の計画に与える影響
- 一方の事業の成功パターンを他方に応用

### 運用ルール

- クロスビジネスの知見は企業レベルの `company-profile.md` に記録する
- 各事業のプロファイルにも「関連する他事業からの示唆」セクションを設ける
- 企業全体の財務制約（借入余力、自己資本比率等）は全事業の計画に反映する

---

## 8. 「事業計画を作りたい」と言われた場合のフロー

ユーザーが「事業計画を作りたい」と発言した場合の標準フロー:

```
Step 1: 企業の確認
  「どの企業（法人/個人事業）の事業計画ですか？」
  └── _registry.md を確認
      ├── 既存企業が見つかった → Step 2 へ
      └── 見つからない → 新規企業を作成 → Step 2 へ

Step 2: 事業の確認
  「どの事業の計画ですか？」
  └── 該当企業の businesses/ を確認
      ├── 既存事業が見つかった → Step 3 へ
      └── 見つからない → 新規事業を作成 → Step 3 へ

Step 3: プロジェクトの確認
  「この事業計画の目的は何ですか？（例: 補助金申請、新規事業、年度計画）」
  └── 該当事業の projects/ を確認
      ├── 同目的の既存PJがある → 継続するか新規作成か確認
      └── ない → 新規プロジェクトを作成

Step 4: セッション開始
  └── _active-context.md を更新
  └── Phase 1（ヒアリング）を開始
```

## 9. 複数ユーザー対応

同一企業に複数のオーナー/担当者がいる場合の運用。

### 基本方針

- 企業レベルの知識（`company-profile.md`）は全ユーザーで共有
- 事業レベルの知識（`business-profile.md`）も共有
- **各ユーザーの `owner-profile.md` は個別に管理**

### `owner-profile.md` に記録する内容

- 氏名・役職
- コミュニケーションスタイル（詳細志向 / 概要志向）
- 意思決定の傾向
- Financial Lens（数字の見方は人によって異なる）
- 関心領域・優先事項

### 運用ルール

- セッション開始時に「どなたとの作業ですか？」を確認（初回以降は `_active-context.md` で推定可能）
- オーナーAが決定した事項をオーナーBのセッションで変更する場合は、必ず確認を取る
- 意思決定の記録には決定者を明記する

```markdown
# decisions/2026-03-28-pricing.md
- 決定事項: 新商品の価格を3,980円に設定
- 決定者: 田中太郎（代表取締役）
- 日付: 2026-03-28
- 背景: ...
```
