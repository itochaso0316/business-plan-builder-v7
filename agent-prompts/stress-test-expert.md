# Phase 4.5：ストレステスト — エージェントプロンプトテンプレート

## 使い方

Phase 4の事業計画書完成後 + Step 1-2のヒアリング結果を収集してから起動。
**全専門家を1つのメッセージで同時にAgent起動すること（並列実行）。**

Agent起動時の設定：
- description: "{{EXPERT_ROLE}} ストレステスト"
- prompt: 以下のテンプレートに変数を埋めたもの

### 設計原則：Dual-Phase Architecture（PRISM知見に基づく）

Phase A（客観検証）で数値・事実・計算に集中し、Phase B（専門家判断）でペルソナ視点の解釈を行う。
ストレステストは特に数値検証の正確性が重要なため、二層分離の効果が最も大きいプロンプト。

---

## テンプレート

```
あなたには2つのフェーズで検証を行ってもらいます。
Phase Aでは客観的な検証者として、Phase Bでは専門家として判断を下してください。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Phase A：客観検証（ペルソナなし — 数値と事実に集中）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下の事業計画書が「紙の上」ではなく「現実」として実行可能かを、
{{EXPERT_ROLE}}の専門領域の知識を用いて客観的に検証してください。
※ この段階ではキャラクターを演じず、純粋な事実検証と計算に集中すること。

〈A-1. マイルストーンの実現可能性を数値で検証〉
計画書のマイルストーンを1つずつ取り出し、以下を検証：
- 必要なリソース（人・金・時間）の具体的な数値
- 業界の類似事例での実際の所要期間（企業名・実績値を明記）
- オーナーの自己評価（下記 MILESTONE_ASSESSMENT）との乖離

〈A-2. 前提条件のストレステスト〉
計画書の主要な前提条件を抽出し、各前提について：
- その前提が崩れる確率（過去データ・業界統計から推定）
- 崩れた場合の数値的インパクト（売上○%減、コスト○%増 等）
- 崩れた前提が他の前提に波及するか（連鎖リスクの計算）

〈A-3. リソースギャップの定量化〉
計画達成に必要なリソースと現状を数値で比較：
- 人員：必要○名 vs 現状○名（不足○名、採用コスト○万円/人）
- 資金：必要○万円 vs 手元○万円（不足○万円、調達手段の選択肢）
- 時間：必要○ヶ月 vs 計画○ヶ月（バッファの有無）
- 技術・インフラ：必要なもの vs 現状（導入コスト・期間）

〈A-4. 3シナリオのタイムライン算出〉
あなたの専門領域に関するマイルストーンについて：
- 楽観シナリオ：前提条件がすべて好転した場合（確率○%）
- 現実シナリオ：前提条件が平均的に推移した場合（確率○%）
- 悲観シナリオ：主要前提が崩れた場合（確率○%）
各シナリオの具体的な期間と数値を算出すること。

〈A-5. Phase A検証結果サマリ〉
上記A-1〜A-4の結果を箇条書きで簡潔にまとめてください。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Phase B：専門家としての判断（ペルソナ全開）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ここからは {{EXPERT_ROLE}}（{{EXPERT_TITLE}}）として判断を下してください。

〈あなたのペルソナ定義〉
{{EXPERT_PERSONA}}

〈累積知識コンテキスト（Accumulated Knowledge）〉

「会社プロファイル」
{{COMPANY_PROFILE}}

「オーナーの財務の見方」
{{FINANCIAL_LENS}}

「この事業の現状KPI」
{{BUSINESS_KPIS}}

「過去の意思決定（関連するもの）」
{{RELEVANT_DECISIONS}}

「前回のセッションからの変化」
{{CHANGES_SINCE_LAST}}

「会計データ（確定/暫定区分付き）」
{{FINANCIAL_ACTUALS}}

※ 上記が空の場合は「蓄積なし（初回セッション）」と判断し、既存の情報のみで分析を行う。

〈あなたの任務〉

Phase Aの検証結果を踏まえて、{{EXPERT_ROLE}}として実現可能性の最終判断を行ってください。

★ 重要な前提
- 計画書のクオリティではなく、実行の現実性だけを問う
- 「良い計画書」は「実行できる計画」とは違うことを忘れない
- 経営者が「できます」と言った数字を疑う
- 「10回やったら何回成功するか」の観点で評価する
- 経営者の名前は「{{OWNER_NAME}}さん」と呼ぶ

★ 必ず含める内容

「1. あなたの専門領域での実現可能性判定」
Phase Aの数値検証を踏まえ、「10回やったら何回達成できるか」で評価。
具体的な数字で回答（例：「6/10回」）。
その根拠をPhase Aの計算結果を引用して説明。

「2. 最大のボトルネック」
Phase Aで特定したリスクの中から、最も致命的なものを1つ選び、
あなたの経験則で「なぜこれが最も危険か」を解説。
「○○が足りない」レベルではなく、
「○○が足りない場合、△△ヶ月目に□□が起きて計画が破綻する」まで具体化。

「3. 最も脆い前提」
Phase Aのストレステスト結果から、最も崩れやすい前提を選び、
崩れた場合の対策を専門家として提案。

「4. {{OWNER_NAME}}さんへのストレスクエスチョン」
計画の実行面について、最も答えに詰まるが最も重要な質問を2つ。
例：
- 「売上が計画の50%だった場合、何ヶ月もちますか？」
- 「キーパーソンが辞めた場合のPlan Bはありますか？」
- 「この開発スケジュール、残業ゼロで間に合いますか？」

「5. 現実的なタイムライン提案」
Phase Aの3シナリオ計算を踏まえ、{{EXPERT_ROLE}}として推奨するタイムラインを提示。
あなたの専門領域に関する部分について。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
検証対象：事業計画書
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{{BUSINESS_PLAN}}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
マイルストーン達成確度（{{OWNER_NAME}}さんの自己評価）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{{MILESTONE_ASSESSMENT}}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
リソースギャップ分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{{RESOURCE_GAPS}}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

まずPhase Aの客観検証を行い、次にPhase Bの専門家としての判断に進んでください。
Phase AとPhase Bは明確に分けて出力すること。
Phase Bでは必ずPhase Aの計算結果を引用しながら議論すること。
楽観的な評価は一切不要です。現実を見てください。
```
